# 캐시(Cache)

캐시는 **DB보다 빠르게 데이터를 조회하기 위해 사용하는 임시 저장소**입니다.  
자주 조회되는 데이터를 메모리에 저장해 두고, DB 접근 횟수를 줄여 **응답 속도를 높이는 것**이 목적입니다.

---

# 캐시를 사용하는 이유

- DB 조회 부하를 줄일 수 있다.
- 자주 요청되는 데이터를 빠르게 반환할 수 있다.
- 전체 서비스 응답 시간을 단축할 수 있다.

즉, 캐시는 **성능 최적화를 위한 계층**이라고 볼 수 있습니다.

---

# 캐시 추가 전략

## 1. Cache-Aside (Lazy Loading)

가장 일반적으로 사용하는 캐시 전략입니다.

### 동작 흐름

- 먼저 캐시를 조회한다.
- 캐시에 데이터가 없으면 DB를 조회한다.
- DB에서 가져온 데이터를 캐시에 저장한 뒤 반환한다.
- 데이터 수정 시에는 DB를 먼저 수정하고, 기존 캐시는 삭제한다.

### 장점

- 필요한 데이터만 캐시에 올라가므로 효율적이다.
- 읽기 성능을 크게 향상시킬 수 있다.
- 구현이 비교적 직관적이다.

### 단점

- 캐시 삭제 직전이나 직후의 짧은 순간에는 **stale data(오래된 데이터)** 가 조회될 수 있다.
- 처음 조회하는 시점에는 반드시 DB를 한 번 거쳐야 한다.

### 무효화 방식

- **Write 시 캐시 삭제**
- 즉, DB를 수정한 뒤 관련 캐시를 제거하여 다음 조회 시 최신 데이터를 다시 적재한다.

### 프로젝트 적용 사례

- **시드 게시물 생성 시** 적용
- 게시글 데이터를 저장한 뒤, 관련 캐시를 삭제하는 방식으로 처리했다.

---

## 2. Write-Behind (Write-Back)

쓰기 성능을 높이기 위한 전략입니다.

### 동작 흐름

- 먼저 캐시에 데이터를 기록한다.
- 이후 일정 시점에 DB에 반영한다.

### 장점

- 쓰기 요청에 매우 빠르게 응답할 수 있다.
- DB에 대한 쓰기 횟수를 줄일 수 있다.

### 단점

- 캐시에만 기록된 상태에서 장애가 발생하면 **데이터 유실 위험**이 있다.
- 구현 난이도가 높고, 장애 복구 전략이 필요하다.

### 무효화 관점

- 이 전략에서는 캐시가 단순 복사본이 아니라 **실제 최신 데이터** 역할을 한다.
- 따라서 DB 반영이 끝난 뒤, 필요에 따라 캐시를 정리하거나 갱신해야 한다.

### 프로젝트 적용 사례

- **게시글 조회 수 증가 테스트**에 적용
- 조회 수처럼 짧은 시간 동안 많이 갱신되는 값에 대해 DB 쓰기 부담을 줄이는 효과를 확인했다.

# 프로젝트에서 구현한 핵심 포인트

## 1. 동시성 제어

- `RLock`을 사용해 공유 저장소 접근 시 기본적인 원자성을 보장했다.
- 여러 요청이 동시에 들어와도 데이터가 꼬이지 않도록 보호했다.

## 2. 만료 처리

- **TTL(Time To Live)** 을 두어 캐시가 일정 시간이 지나면 만료되도록 했다.
- 실제 삭제는 조회 시점에 수행하는 **Lazy Expiration** 방식으로 처리했다.

## 3. 외부 사용성

- FastAPI로 API를 제공하여 외부에서 캐시 서버처럼 사용할 수 있도록 구성했다.

## 4. 무효화 방식

- Cache-Aside 전략에서는 **데이터 수정 후 캐시 삭제**를 적용했다.
- 이를 통해 다음 조회에서 최신 데이터가 다시 적재되도록 만들었다.

# 트러블 슈팅

## 문제 인식

처음에는  
“레디스처럼 빠른 시스템을 만들고 있으니, 단일 스레드여도 무조건 매우 빨라야 한다”  
라고 생각했다.

하지만 실제로는 단일 스레드 구조에서 여러 요청이 몰리면,  
동시에 하나의 저장소를 순차적으로 처리해야 하므로 **대기 시간**이 생길 수밖에 없었다.

## 원인

- 단일 스레드 구조에서는 한 번에 하나의 작업만 처리할 수 있다.
- 따라서 요청이 많아질수록 처리 지연이 발생한다.
- 즉, “단일 스레드 = 무조건 느림”은 아니지만,  
	“병렬 처리 한계가 있으므로 부하가 커지면 지연이 생길 수 있다”는 점을 확인했다.

## 배운 점

- 캐시 시스템의 성능은 단순히 “메모리를 쓴다”는 이유만으로 결정되지 않는다.
- **동시성 구조, 락 방식, 만료 처리 방식, 쓰기 전략**까지 함께 고려해야 한다.
- 결국 캐시는 “빠른 저장소”라기보다,  
    **적절한 전략으로 DB 부하를 줄이고 응답 속도를 개선하는 시스템**이라고 이해하게 되었다.
